# Payroll

Distribute USDC or USDT to multiple recipients efficiently. Choose between single-transaction batch payments or privacy-enhanced distribution that fans out through independent routing.

## Execution Modes

### Wallet Mode

Single transaction with multiple recipients:

<Steps>
  <Step title="Add Recipients">
    Enter up to 12 recipient addresses and amounts. Each row is validated independently.
  </Step>
  <Step title="Review Batch">
    Preview the complete transaction with total amount and fees.
  </Step>
  <Step title="Sign Once">
    Single signature covers all transfers in the batch.
  </Step>
  <Step title="Execute">
    One signed transaction, one Solscan link on success.
  </Step>
</Steps>

**Transaction Structure:**
```typescript
// Each recipient becomes an instruction
const instructions = recipients.map(recipient => [
  // Create ATA if needed (idempotent)
  createAssociatedTokenAccountIdempotentInstruction(
    payer.publicKey,
    recipient.ata,
    recipient.address,
    mint
  ),
  // Transfer tokens
  createTransferCheckedInstruction(
    payerATA,
    mint,
    recipient.ata,
    payer.publicKey,
    recipient.amount,
    decimals
  )
]);
```

### Private Mode

Independent private transfers to each recipient:

<Steps>
  <Step title="Add Recipients">
    Same interface as wallet mode, but each recipient gets independent privacy.
  </Step>
  <Step title="Generate Deposit Addresses">
    Routing layer provides N ephemeral addresses for N recipients.
  </Step>
  <Step title="Fund Deposits">
    You send the total amount, which gets distributed across ephemeral addresses.
  </Step>
  <Step title="Track Distribution">
    Each recipient's transfer is tracked independently in the status stepper.
  </Step>
</Steps>

**Privacy Benefits:**
- Each recipient receives funds from a different routing address
- No single transaction links you to all recipients
- Recipients cannot see other recipients in the batch

## Recipient Management

### Adding Recipients

| Field | Validation | Required |
|-------|------------|----------|
| **Address** | Valid Solana address (base58, 32-44 chars) | ✅ |
| **Amount** | Positive number, sufficient balance | ✅ |
| **Label** | Optional description for your records | ❌ |

### Validation Per Row

```typescript
const validateRecipient = (recipient) => {
  // Address validation
  try {
    new PublicKey(recipient.address);
  } catch {
    return "Invalid Solana address";
  }
  
  // Amount validation
  if (recipient.amount <= 0) {
    return "Amount must be positive";
  }
  
  // Token support validation
  if (!['USDC', 'USDT'].includes(recipient.token)) {
    return "Only USDC and USDT supported";
  }
  
  return null; // Valid
};
```

### Batch Limits

| Mode | Max Recipients | Transaction Limit |
|------|----------------|-------------------|
| **Wallet Mode** | 12 | Solana 1232-byte limit |
| **Private Mode** | No limit | Scales linearly in routing calls |

<Note>
Wallet mode is capped at 12 recipients to stay within Solana's transaction size limit. Private mode has no per-batch cap but each recipient requires a separate routing call.
</Note>

## Amount Helpers

Distribute amounts efficiently:

| Helper | Function |
|--------|----------|
| **Equal Split** | Divide total amount equally among all recipients |
| **Percentage** | Allocate based on percentage weights |
| **Remaining** | Auto-calculate final recipient to use all funds |

### Equal Split Example

```typescript
// Split 1000 USDC among 5 recipients
const totalAmount = 1000;
const recipientCount = 5;
const amountPerRecipient = totalAmount / recipientCount; // 200 USDC each
```

## Status Tracking

### Wallet Mode

Single transaction status:
- **Pending**: Transaction submitted to network
- **Confirming**: Awaiting block confirmation
- **Success**: All recipients received funds
- **Failed**: Transaction reverted, funds returned

### Private Mode

Per-recipient tracking:
- **Queued**: Waiting for routing layer processing
- **Processing**: Individual transfer in progress  
- **Sent**: Recipient's transfer completed
- **Failed**: Individual transfer failed, will retry

## Fees

| Mode | Fee Structure |
|------|---------------|
| **Wallet Mode** | Network fees only (~0.000005 SOL per instruction) |
| **Private Mode** | Network fees + routing fee per recipient |

### Fee Calculation

```typescript
// Wallet mode: Base fee + (recipients * instruction_fee)
const walletModeFee = 0.000005 + (recipients.length * 0.000005);

// Private mode: Base fee + (recipients * routing_fee)  
const privateModeFee = 0.000005 + (recipients.length * routingFee);
```

## ATA Management

### Automatic ATA Creation

All payroll transactions include idempotent Associated Token Account (ATA) creation:

```typescript
// Creates ATA only if it doesn't exist
createAssociatedTokenAccountIdempotentInstruction(
  payer.publicKey,    // Fee payer
  recipientATA,       // ATA address
  recipient.address,  // Owner
  mint               // Token mint
);
```

**Benefits:**
- Recipients don't need to pre-create token accounts
- No failed transactions due to missing ATAs
- Idempotent: safe to call multiple times

## Troubleshooting

### Common Issues

| Error | Cause | Solution |
|-------|-------|----------|
| "Insufficient balance" | Not enough tokens for total + fees | Check total includes all fees |
| "Invalid recipient" | Malformed address in batch | Validate all addresses |
| "Transaction too large" | > 12 recipients in wallet mode | Use private mode or split batch |
| "ATA creation failed" | Network congestion | Retry with higher priority fee |

### Best Practices

- **Test small batches**: Start with 2-3 recipients to verify setup
- **Verify addresses**: Double-check all recipient addresses
- **Monitor balances**: Ensure sufficient funds for total + fees
- **Keep records**: Export transaction data for accounting

## Security Considerations

<Warning>
**Address Verification**: Always verify recipient addresses before executing payroll. Incorrect addresses result in permanent loss of funds.
</Warning>

### Validation Checklist

- [ ] All addresses are valid Solana addresses
- [ ] Amounts are positive and correctly formatted
- [ ] Total amount + fees ≤ available balance
- [ ] Recipients are intended payees
- [ ] Token type matches intended distribution
